## 3. 产品规范缺口（本次测试暴露）

### Gap-1: /ingest 不接受 URL

**证据**: 用户想 ingest `https://x.com/garrytan/status/2042925773300908103`，规范第 1 步硬要求路径在 raw/ 下。导致用户必须自己**把推文粘贴到对话 → Claude Write 到 raw/ → 再 /ingest**。三步。

**严重度**: 中。第一次 ingest（forrestchang CLAUDE.md）时也遇到，当时用 curl 绕过了；本次 x.com 反爬无法 curl，摩擦感放大。

**候选方案**:
- **A**. 改 `.claude/commands/ingest.md` 第 1 步：若 `$ARGUMENTS` 以 `http(s)://` 开头，调用 pre-flight 抓取子流程（WebFetch → $B → 让用户粘贴 fallback）。
- **B**. 新增 `/clip <url>` 命令，专职 URL→raw，保持 /ingest 职责单一。**契合刚 ingest 完的 [[purpose-built-tooling]] 原则**。
- **C**. 文档化：CLAUDE.md 加一节"ingest URL 资料手动流程"。零代码改。

**推荐**: **B**——dogfood Tan 的 "thin + narrow tooling" 原则。

### Gap-2: raw/ 写入权规则过严

**证据**: CLAUDE.md 写"只有用户才能修改 raw/"，但实际测试中 AI 两次都在用户授权下写入 raw/（第一次 curl 保存 forrestchang，第二次用户粘贴后 AI Write）。规则与实际行为冲突。

**建议修订**:

```markdown
- `raw/` — 只读真相层
  - 你可以 Read，**绝不能** 修改已存在的 raw/ 文件
  - 在用户明确授权下（例如触发 /ingest URL 或 /clip），可以**新建** raw/ 文件，但需带 source_url + fetched_at frontmatter
```

### Gap-3: x.com 抓取链路未规范

**证据**: 头 less 浏览器无登录遇登录墙；cookie 导入未规范；无 fallback 引导。用户只能"自己粘贴"。

**候选**: 在 `/clip` 规范（若采纳 Gap-1 方案 B）里明示抓取优先级：
1. WebFetch（适合静态 HTML）
2. gstack `$B goto + text`（JS SPA）
3. `$B cookie-import-browser` 后重试（需要登录的页面）
4. AskUserQuestion 让用户粘贴（最终 fallback）

### Gap-4: ingest 的产物没有统一入口供 lint / query 发现

**证据**: wiki/lint-reports 和 output/test-reports 目前散落。没有一个"本次 ingest 做了什么"的 manifest 文件。若 /lint 需要复查"最近一次 ingest 的所有产物"，只能 grep log.md。

**建议**: `/ingest` 完成时附加输出 `wiki/ingest-reports/<date>-<slug>.md`，作为"本次连锁的清单" manifest。或加强 log.md 格式使其更结构化。优先级低。
